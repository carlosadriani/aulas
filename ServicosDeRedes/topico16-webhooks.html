16. Webhooks e Eventos

/span>

Compreensão profunda do modelo de comunicação orientada a eventos na Web através do fluxo inverso do protocolo HTTP, focando na otimização de infraestrutura e tráfego de rede.

01. O que são Webhooks?

Um Webhook é uma mecânica de comunicação assíncrona entre sistemas baseada no padrão de Inversão de Controlo (IoC). Em vez de um cliente perguntar periodicamente se uma informação mudou, o servidor assume a responsabilidade de notificar ativamente o cliente no exato momento em que um evento interno relevante acontece.

Perspetiva de Redes: Em termos de infraestrutura, um Webhook transforma o servidor original em cliente e o cliente original em servidor. O sistema recetor precisa de expor um endpoint público na Camada de Aplicação (HTTP/HTTPS) para escutar e processar as requisições recebidas[cite: 8, 29].

02. Análise de Eficiência: Polling vs. Webhook

Para entender o impacto de rede e desempenho de software, compara-se o modelo tradicional de requisição-resposta repetitiva (Polling) com o modelo baseado em gatilhos orientados a eventos (Webhooks):

Critério Técnico Short/Long Polling Webhook (Event-Driven)
Consumo de Banda Muito Alto (Múltiplos headers HTTP vazios trafegados) Mínimo (Apenas trafega dados se o evento ocorrer)
Direção do Fluxo Cliente → Servidor Servidor → URL do Cliente (Inverso)
Latência do Dado Depende do intervalo de verificação configurado Próxima de zero (Tempo real / Real-time)
Infraestrutura Fácil (Cliente gerencia as chamadas em loop) Exige que o cliente mantenha uma porta exposta à Internet

03. Anatomia de um Webhook

Os Webhooks operam quase em exclusividade enviando requisições com o método HTTP POST. O corpo da mensagem transporta metadados estruturados (geralmente JSON) identificando a entidade e a ação que disparou o gatilho[cite: 23, 29].

Exemplo real de Cabeçalhos e Payload JSON enviados por um Webhook (Ex: Gateway de Pagamento):

POST /webhooks/receber-pagamento HTTP/1.1
Host: api.seuapp.com
Content-Type: application/json
X-Webhook-Event: payment.captured
X-Webhook-Signature: t=1672531199,v1=7a8b9c...

{
  "event_id": "evt_90213812",

04. Resiliência, Retries e Idempotência

A comunicação através de redes públicas está sujeita a falhas temporárias (latência alta, perdas de pacotes ou instabilidades do servidor recetor). Por esse motivo, sistemas robustos que emitem Webhooks adotam práticas estritas de engenharia de confiabilidade:

  • Políticas de Retry (Retentativas): Quando o recetor não responde com um código HTTP de sucesso (`2xx`), o emissor tenta enviar o payload novamente usando algoritmos de Exponential Backoff (afastamento exponencial) com jitter, evitando um ataque de negação de serviço (DoS) involuntário no cliente[cite: 20].
  • Dead Letter Queue (DLQ): Se um número limite de tentativas falhar consecutivamente (ex: 5 vezes), a mensagem é movida para uma fila de falhas estruturada para auditoria manual.
  • Idempotência Obrigatória no Recetor: Como pacotes podem sofrer duplicidade na rede, o recetor do Webhook deve validar o campo `event_id` no banco de dados antes de processar as regras de negócio, garantindo que o mesmo evento nunca seja processado duas vezes.

05. Segurança Avançada em Webhooks

Manter um endpoint público exposto à Internet para receber dados acarreta riscos críticos de segurança (como injeção de payloads maliciosos oriundos de atacantes fingindo ser o servidor legítimo).

A Solução: Assinaturas Criptográficas com Segredo Partilhado (HMAC)

Para mitigar ataques de personificação, o emissor e o recetor partilham previamente uma chave secreta (Secret Key). O processo de validação baseia-se nas seguintes etapas:

  1. O emissor concatena o timestamp da requisição com o corpo bruto da mensagem (JSON)[cite: 23].
  2. Gera um hash hash hash criptográfico utilizando o algoritmo **HMAC-SHA256** com a chave secreta.
  3. Envia esse hash no cabeçalho customizado da requisição (ex: `X-Webhook-Signature`).
  4. Ao receber a requisição, o recetor recalcula localmente o hash com os mesmos dados recebidos. Se os valores baterem, a autenticidade e integridade do pacote estão validadas.
