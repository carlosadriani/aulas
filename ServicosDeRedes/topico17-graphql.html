17. GraphQL e Otimização de Consultas na Web

Estudo detalhado do GraphQL como linguagem de consulta e ambiente de execução de dados. Compreensão de como a modelagem orientada a grafos soluciona ineficiências de banda e requisições repetitivas na rede.

01. O que é o GraphQL?

Criado pelo Facebook em 2012 e disponibilizado como código aberto em 2015, o GraphQL é uma especificação para uma linguagem de consulta de dados (Query Language) combinada com um runtime no lado do servidor para responder a essas consultas através de um sistema de tipagem estrita.

Perspetiva de Arquitetura de Redes: Diferente do REST que distribui o sistema em dezenas de URIs/endpoints expostos, o GraphQL centraliza toda a comunicação em um único endpoint HTTP (geralmente `POST /graphql`). O cliente submete a estrutura exata do grafo de dados que deseja e o servidor responde em um único pacote consolidado.

02. Eficiência de Rede: Over-fetching e Under-fetching

Em ambientes móveis ou redes de baixa velocidade, as restrições nativas do REST costumam criar desperdício de infraestrutura através de dois problemas clássicos:

Over-fetching (Sobrecarga de Dados)

Ocorre quando o endpoint do servidor retorna mais dados do que o cliente realmente necessita para renderizar a interface atual.

Exemplo: Chamar `GET /usuarios/1` apenas para mostrar o nome na tela, mas o payload JSON vir acompanhado de endereços, histórico de compras, tokens e preferências, desperdiçando pacotes na rede.

Under-fetching (Múltiplas Requisições)

Ocorre quando um único endpoint não possui dados suficientes, forçando a aplicação cliente a disparar requisições HTTP sequenciais em cascata (problema do N+1).

Exemplo: Chamar `/usuarios` para ler a lista, depois `/usuarios/1/posts` para ler os posts e, por fim, `/posts/99/comentarios`, gerando alta latência de handshakes na rede.

03. Anatomia da Consulta e Resposta

O GraphQL transfere o controle de previsibilidade para o cliente. A estrutura da requisição espelha perfeitamente o formato da resposta JSON.

Requisição do Cliente (Query Body):

query {
  usuario(id: "42") {
    nome
    email
    posts {
      titulo
    }
  }
}

Resposta Exata do Servidor (JSON):

{
  "data": {
    "usuario": {
      "nome": "Carlos",
      "email": "carlos@edu.br",
      "posts": [
        {"titulo": "Introdução ao TCP/IP"}
      ]
    }
  }
}

04. O Sistema de Tipagem e Operações

No backend, define-se estritamente um Schema utilizando a *Schema Definition Language (SDL)*. Este contrato prevê três tipos principais de operações raíz que mapeiam as intenções da camada de aplicação:

Operação Propósito Técnico Equivalente no REST
Query Leitura e recuperação de dados isolados ou encadeados. Não gera efeitos colaterais no servidor. GET
Mutation Operações de escrita que alteram o estado da aplicação (Criação, Atualização, Remoção). POST PUT DELETE
Subscription Conexão persistente em tempo real. O servidor envia dados ao cliente continuamente assim que eventos acontecem. WebSockets / SSE

05. Impacto de Infraestrutura e Redes

Apesar dos grandes benefícios na economia de tráfego de banda, a implementação do GraphQL impõe novos desafios técnicos para as equipas de infraestrutura e engenharia de redes:

  • Dificuldade de Caching HTTP: Como o GraphQL utiliza majoritariamente requisições POST direcionadas a um único endpoint, mecanismos nativos de cache web baseados em URIs (como proxies reversos, CDNs, Varnish ou o próprio cache do navegador) deixam de funcionar automaticamente. A estratégia de cache deve ser deslocada para o nível de aplicação ou persistência interna.
  • Risco de Ataques de Complexidade (DoS): Como o cliente define o formato da árvore de consulta, um atacante pode injetar uma query com recursividade infinita ou níveis profundos de aninhamento de grafos (ex: usuário que puxa posts, que puxa comentários, que puxa usuário do comentário, e assim por diante), derrubando os recursos de processamento computacional e memória do servidor.
    Solução: Implementar algoritmos de análise de profundidade máxima de consulta (Query Depth Limiting) ou cálculo de custo de query.
